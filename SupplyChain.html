Supply Chain Latency & Risk Prediction

Project Overview: This project develops an end-to-end machine learning system to predict shipment delivery outcomes across a global supply chain network. Using the DataCo Smart Supply Chain dataset (180K+ orders), the system classifies each order into one of four outcomes: Advance Shipping, On Time, Late Delivery, or Canceled.
The pipeline covers the full data science lifecycle: from data cleaning and exploratory analysis, through unsupervised learning (PCA, UMAP, KMeans clustering), to supervised modeling with LightGBM and XGBoost.

Tools Used: Python, Pandas, NumPy, Matplotlib, Seaborn, Scikit-learn, XGBoost, LightGBM, Imblearn, Streamlit

Problem Foundation & Dataset Overview

Problem Description & Proposed Solution


Problem Description: Late and canceled shipments are a silent killer in supply chains — they cause stockouts, unhappy customers, and wasted resources. Most logistics systems only flag problems after they happen. This project flips that by predicting delivery outcomes before shipment, giving teams a chance to act early before things go wrong.

Proposed Solution: In this project, we focus on creating a predictive model that tries to predict whether a trip will:

  • Arrive Earlier than Expected
  • Arrive Later than Expected
  • Arrive On Time
  • Cancelled (Challenging to Predict)

DataSet Overview


The DataCo Supply Chain Dataset have been used for this project, the dataset included 3 tables:

Supply Chain Trips Data: The main table in the project, includes 180K records and 53 features, each record is about the summary of a delivery trip such as estimated delivery time (in days), delivery trip start time, where is the trip starting from & going to, our target feature (Delivery Status) and more

Features Descriptions: A table including the explanation of each feature in the main dataset, this table was only used for the business understanding phase

Tokenized Access Logs: This table included redundant & non relevant features about each transaction, because the features of this table were not any useful, it was not used

Milestone 1: Data Exploration & Preprocessing

1. Data Cleaning & Preprocessing


After inspecting the dataset for nulls, duplicates, and data types — none were found. Cleaning focused on four areas:

Removed PII & ID Features: Customer identifiers, order/product IDs, and static metadata (images, descriptions) were dropped as they carry no analytical value.

Removed Leakage Features: Days for shipping (real), Late_delivery_risk, and Order Status were dropped — all three are only known after delivery, making them illegal inputs for a prediction model.

Removed Redundant Features: Duplicate columns were identified and dropped — profit, price, and sales columns each had an exact duplicate. Geographic columns Order City and Order State were also removed since Order Region and Country already capture that information.

Handled Extreme Outliers: Three filters removed impossible values — discount above 450, benefit below -2,900, and sales above 550.

Dataset went from 180,519 rows × 53 columns to 180,031 rows × 30 columns.

2. Exploratory Data Analysis (EDA)


After cleaning the dataset, EDA steps were performed to understand the relationships between the remaining 29 features & our target columns, alongside with the relation between numerical columns to each others. Key insights included:
  • The majority of delivery trips were classified as Late Delivery, while only a small proportion of trips were cancelled, This means that target features are imbalanced & balancing techniques should be performed
  • Despite being the slowest tier, Standard Class accounts for nearly all advance shipments; suggesting sellers set conservative delivery windows for standard orders, making early arrival easy
  • High-volume cities like Los Angeles and New York show moderate late delivery rates (~50-60%), while several low-volume cities across Texas and the Midwest show alarmingly high rates of 70-80%: a pattern that traditional volume-based monitoring would completely overlook
  • Late deliveries consistently dominate across all time dimensions: by year, month, day, and hour. Maintaining the highest count at every point, confirming it's a systemic issue rather than a seasonal one
  • The sharp drop in all shipment counts toward 2018 is a data collection artifact, not a real business trend: 2018 is severely underrepresented in the dataset and should be interpreted with caution

3. Feature Engineering & Unsupervised Learning


Feature Engineering: The shipping date was decomposed into month, day of week, and hour. Two binary flags were created from high-signal categorical columns — Is_Transfer and Is_Standard_Class / Is_Second_Class. Categorical columns were then encoded using Target Encoding for high-cardinality features and One-Hot Encoding for the rest.

Unsupervised Learning: Before supervised modeling, unsupervised techniques were applied to explore the dataset's natural structure and validate whether the four delivery outcomes form distinguishable patterns.

PCA reduced the feature space down to 40 components while retaining 88.7% of the variance, compressing the data for efficient UMAP processing.

UMAP projected the data into 2D, revealing that delivery outcomes form partially separable regions — confirming that the features carry real predictive signal, particularly Shipping Canceled which formed a tight isolated cluster.

KMeans (k=4) was overlaid on the UMAP projection to see how naturally forming groups align with actual delivery labels, showing reasonable correspondence with the true outcome distribution.

Milestone 2: Data Modeling

Data Modeling Overview


After preparing the data and ensuring its quality & consistency, LightGBM and XGBoost models were trained on the cleaned dataset. But before modeling, the target column was imbalanced with the next counts:
  • Late Shipment: about 79K targets out of 144k training values belonged to this class
  • Advanced Shipment: 33.1K targets belonged to this class
  • Late Shipment: 25.7k targets belonged to this class
  • Shipment Canceled: Only 6.2k targets belonged to this class
So, Undersampling & SMOTE were applied to balance all target classes to be at 60K, then both models were trained

1. LightGBM

3. Model Selection


After comparing both models across all metrics, XGBoost outperformed LightGBM on every measure — higher accuracy (73.7% vs 72%), better macro F1 (0.65 vs 0.60), and stronger precision/recall across all four classes. XGBoost was therefore selected as the final model for deployment.

Milestone 3: Model Deployment

Prediction System Application


After modeling, the model was deployed on a simple 1 page application that forecasts whether the shipment will arrive on time, arrive late, arrive early or be canceled by giving it order details, customer details, financial details & shipping date.
